## 🧠 Core Logic

### Backtest Engine (`backtest.py`)

The main duty of this module is to simulate trades using historical market data, based on the technical indicators and trading signals generated by the `signals.py` module, and then calculate core metrics based on `equity` and `return` to assess the quality and performance of the strategy.

In this module, I input a dataframe containing multiple indicators such as `closing_price`, `opening_price`, and trading signals including `golden_cross`, `death_cross` generated by the `signals.py` module. It then outputs a Python dictionary containing 5 core metrics: `total_return`, `max_drawdown`, `win_rate`, `profit_loss_ratio`, and `num_trades`.

Finally, I can systematically evaluate the quality and performance of this trading strategy using quantitative finance knowledge and evaluation criteria, combined with the calculated core metrics.

---

#### `run_backtest(df, commission=0.00075) → df`

The main duty of this function is to simulate real trades using historical market data and calculate core indicators `equity_curve` and `daily_return`.

**Design decisions:**

- **`ffill` state machine, not `cumsum`**  
  **Why:**  
  Initially, I adopted the `cumsum` method to simulate the states of `open_position`, `hold_position`, `close_position`, and `keep_empty_position`, combined with the `.clip()` function to confine the `position` field to [0, 1].

  But afterwards, I found there is a boundary case that this method cannot cover well. For instance, when the first signal is `death_cross(-1)`, the cumulative sum will become -1, and then the `.clip()` function will clip -1 to 0 and assign 0 to the `position` field. Now the cumulative sum remains -1. Subsequently, if the next signal is `golden_cross(+1)`, the cumulative sum will become 0 based on the previous value of -1, and then the `.clip()` function will clip 0 to 0 and assign 0 to the `position` field. But in our model, a `golden_cross` signal is the signal to open a position, where the `position` field should be 1.

  Beyond the boundary case, the root issue is semantic mismatch: `cumsum` carries "incremental accumulation" semantics, while a trading position is a continuous state. `ffill` ("forward fill the latest signal") naturally matches state semantics — the right tool for the job, not a patch on the wrong one.

  So ultimately, I adopted a state machine model that better suits such situations and can reuse the previous code logic (the series of `golden_cross` and `death_cross` signals) to set states. This clarifies the code logic and reduces scenario complexity. Most importantly, the state machine model avoids boundary cases more effectively and easily.

  **How:**
  ```python
  target_position[df["golden_cross"]] = 1.0
  target_position[df["death_cross"]] = 0.0
  target_position = target_position.ffill().fillna(0)
  ```
  Based on the previously generated dataframe, let `1.0` denote the `open_position` and `hold_position` states, and let `0.0` denote the `close_position` and `keep_empty_position` states.

- **Double `shift(1)` to avoid Look-Ahead Bias**  
  **Why:**  
  We can only see signals after T-day market close, so I use the `.shift()` function to shift the opening date to T+1. And because I adopted the `T+1 close-price` model, which assumes I open a position on T+1 at the closing price and bear the full price fluctuation on T+2, I subsequently use a second `.shift()` function to ensure I only earn returns and accrue equity on T+2.

  **How:**
  ```python
  df["position"] = target_position.shift(1).fillna(0).astype(int)
  daily_pct_change = df["收盘"].pct_change().fillna(0)
  df["daily_return"] = df["position"].shift(1).fillna(0) * daily_pct_change
  ```
  I use the first `.shift()` to ensure the opening date is T+1, then use `.pct_change()` to calculate the percentage change in closing prices, and finally use the second `.shift()` to ensure I only earn returns on the day after T+1.

- **T+1 close-price execution model**  
  **Why:**  
  The `T+1 close-price` model I adopted is more conservative and simplified than mainstream industry models. For instance, it assumes I open a position at the T+1 closing price on T+1 and bear the full price swing on that closing date, whereas the more reasonable approach would be to buy and sell earlier based on signals that emerge on T day.

  **Note the asymmetry inherent in this model:**
  - **Entry day (T+1)**: position becomes 1 at market close — no price fluctuation captured (`daily_return = 0 × pct_change - commission = -commission`)
  - **Holding period (T+2 onwards)**: full price fluctuation captured
  - **Exit day**: full day's fluctuation captured before position zeroes out (`daily_return = 1 × pct_change - commission`)

  This asymmetry is why the entry baseline requires a previous-day shift (see `prev_open_iloc` in `calculate_metrics`) but the exit baseline does not.

- **Single-side commission `0.00075`**  
  **Why:**  
  Typically, the total commission for one complete round-trip trade in the market is approximately 0.15%, so based on this, I divided it into two equal single-side commissions applied separately to open and close positions.

  **How:**
  ```python
  position_change = df["position"].diff().abs().fillna(0)
  df["daily_return"] -= position_change * commission
  ```
  Identify position change days and deduct commission on each trading date.

#### `calculate_metrics(df) → dict`

The main duty of this function is to calculate core metrics to assess the performance and quality of the strategy, based on the dataframe returned by the previous function.

**Implementation detail: single-trade return calculation**
- **Single-trade return uses `equity_curve[open_date - 1]` as entry baseline**  
  **Why:** Commission is deducted on the `position.diff()` day, so `equity_curve[open_date]` already reflects the deducted entry commission. Using it as entry baseline would cancel out the entry commission and systematically underestimate trading cost (which biases `profit_loss_ratio` toward false optimism). Using the previous day's equity captures the full round-trip cost (entry commission + holding P&L + exit commission).

**Five metrics, why each matters:**
These five metrics are the most common evaluation criteria in the mainstream industry.

- **`total_return`** — overall return rate over the backtest window.  
  This metric represents the total returns generated from the first transaction using this strategy.

- **`max_drawdown`** — the maximum drawdown from the historical highest equity to current equity.  
  This is the most important metric, more so than `total_return`, because this metric shows the deepest drawdown I would have to endure, which is highly related to mental sustainability and the risk of catastrophic concentrated losses.

- **`win_rate`** — winning trades / total trades (per-trade, not per-day).  
  This metric represents the percentage of complete round-trip trades that are profitable.

- **`profit_loss_ratio`** — average win / |average loss|.  
  Combined with the `win_rate` metric, it measures how big the wins are relative to the losses — not directly a "risk-reward ratio" (which would be Sharpe / Calmar / Sortino). Is this strategy worthy? Does it perform well? Typically, we can tolerate low `win_rate` when `profit_loss_ratio` is high. Additionally, a strategy with low win rate AND low profit-loss ratio is garbage.

  In this MA-cross backtest on 000001.SZ, the strategy showed `win_rate` = 22.22% and `profit_loss_ratio` = 0.44 — a textbook "low win rate + low profit-loss ratio = garbage profile". Root cause: MA-cross is inherently trend-following, but the market regime in 2025-2026 for this stock was range-bound with a downward drift — strategy-market regime mismatch.

- **`num_trades`** — frequency of trading.  
  This metric counts the total number of complete round-trip trades.

### Moving Average Cross Strategy

A **golden cross** occurs when the short-term MA crosses **above** the long-term MA — traditionally interpreted as a bullish signal. A **death cross** is the opposite.

The key insight: **a cross is a state change, not a state**. The detection compares today's relationship between MA5 and MA20 with yesterday's:

```python
df["golden_cross"] = (
    (short_ma > long_ma) &
    (short_ma.shift(1) <= long_ma.shift(1))
)
```

### Local Cache Layer

To decouple development from network dependency (especially when working across borders), data fetching is wrapped in a cache layer:
- First run: fetches from tushare, saves to local CSV
- Subsequent runs: reads directly from CSV (no network needed)
- Use `force_refresh=True` to bypass cache

### Volume Subplot

Price-only signals are noisy. Adding volume as a secondary indicator helps validate moves:
- **Volume rising with price** → likely a real trend
- **Volume falling while price rises** → likely a fake breakout

The volume subplot uses `plt.subplots()` with `sharex=True` to keep both panels synchronized.

## ⚠️ Known Limitations

- **MA cross signals are weak indicators**. They generate frequent false signals in sideways markets and lag in trending markets. This project is for learning purposes — **do not use it for actual trading**.
- The data source `tushare` is suitable for personal research only. Production-grade quantitative systems use Wind, level-2 feeds, or proprietary data sources.
- Currently only supports a single stock at a time. Multi-stock support, backtesting, and risk metrics (Sharpe, max drawdown, win rate) are planned for future versions.
- Backtesting currently uses unadjusted price data due to free-tier limitations of Tushare. Forward-adjusted (qfq) data is the standard correct choice for backtesting, eliminating price distortion from abrupt gaps on ex-dividend and ex-rights dates. Data migration to forward-adjusted pricing is scheduled after upgrading to a paid subscription tier.
